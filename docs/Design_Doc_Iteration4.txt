1.3 Use Case Descriptions — Iteration 4 Add-ons
-------------------------------------------------

Use Case Number:
  UC-10
Use Case Name:
  GDPR Consent Management
Overview:
  Before any medical data is accessed or stored, the system presents
  a GDPR-compliant consent modal allowing the patient to explicitly
  opt in or out of (a) EMR record access and (b) conversation history
  storage. Consent can be granted at session start and updated at any
  time during the session, fulfilling GDPR Art. 5(1)(a) (Lawfulness &
  Transparency) and Art. 7 (Conditions for Consent).
Actors:
  Patient
Pre condition:
  The patient must navigate to the /chat page. A new session must be
  created or an existing session must be active.
Flow:
  Main (success) Flow:
  1. The patient navigates to the chat page.
  2. The system detects no active session and displays the GDPR
     Consent Modal (ConsentModal component).
  3. The modal presents:
     a. An AI disclaimer — "Robert is an AI assistant, not a
        licensed physician" (GDPR Art. 22).
     b. EMR Consent checkbox — "Allow access to my Electronic
        Medical Records" (Art. 5(1)(a)).
     c. History Storage checkbox — "Store this conversation for
        future reference" (Art. 5(1)(e)).
     d. Data rights notice — right to delete via sidebar,
        no data used for AI training.
  4. The patient checks/unchecks preferences and clicks "Continue."
  5. The system creates a new session (POST /sessions/{patient_id})
     with the selected consent flags and a 30-day expiry.
  6. The chat interface loads with the chosen permissions.
  Alternate Flow:
  4a. If the patient clicks "Continue without data sharing," the
      session is created with both consent flags set to false.
      The chatbot provides only general medical advice (no EMR access)
      and messages are ephemeral (lost on refresh).
  Update Flow (mid-session):
  1. The system sends PUT /gdpr/consent/{session_id} with updated flags.
  2. If store_history_consent is revoked, all stored messages in the
     session are immediately purged (GDPR Art. 17).
Post Condition:
  The session is created with explicit consent preferences recorded.
  All subsequent processing respects these flags — no EMR data is
  accessed without emr_consent=true, no messages are persisted without
  store_history_consent=true.

--------------------------------------------------------------------------------

Use Case Number:
  UC-11
Use Case Name:
  Session Management & Retention
Overview:
  The system manages persistent chat sessions with automatic retention
  policies. Each session has a 30-day time-to-live (TTL) after which
  it is automatically deleted, ensuring compliance with GDPR Art. 5(1)(e)
  — Storage Limitation. Patients can create, list, switch between, and
  resume previous conversations.
Actors:
  Patient, System (automatic cleanup)
Pre condition:
  The patient must have completed GDPR consent (UC-10) at least once.
  The backend must be running (retention cleanup executes at startup).
Flow:
  Main (success) Flow — Create Session:
  1. The patient opens the chat page and completes consent (UC-10).
  2. The system generates a UUID, sets created_at and expires_at
     (now + 30 days), and saves the session as a JSON file.
  3. The session appears in the sidebar.
  Main (success) Flow — List & Switch Sessions:
  1. The patient clicks on a previous session in the sidebar.
  2. The system loads GET /sessions/{session_id} and displays
     the stored conversation history.
  3. The patient can continue the conversation from where they left off.
  Automatic Retention Cleanup:
  1. At server startup, the system calls run_retention_cleanup().
  2. All sessions where expires_at < current time are permanently
     deleted from disk.
  3. This runs once per application lifecycle (server restart).
Post Condition:
  Active sessions are available for up to 30 days. Expired sessions
  are automatically purged without user intervention. Session files
  are stored as JSON on disk at data/sessions/{uuid}.json.

Use Case Number:
  UC-12
Use Case Name:
  Right to Erasure (GDPR Art. 17)
Overview:
  The patient can permanently delete individual sessions or all sessions
  associated with their patient ID, exercising their GDPR Art. 17 Right
  to Erasure. The system physically removes the session files from disk
  and returns a compliance receipt with the GDPR article reference and
  deletion timestamp.
Actors:
  Patient
Pre condition:
  At least one session must exist for the patient. The patient must be
  on the chat page with the sidebar visible.
Flow:
  Main (success) Flow — Single Session Deletion:
  1. The patient clicks the 🗑 delete button next to a session in the
     sidebar.
  2. The frontend sends DELETE /gdpr/sessions/{session_id}.
  3. The system locates the session file on disk.
  4. The system permanently deletes the file (os.remove()).
  5. The system returns a JSON receipt:
     { gdpr_action: "session_deleted",
       gdpr_article: "Art. 17 — Right to Erasure",
       deleted_at: "2026-04-19T08:00:00Z" }
  6. The session is removed from the sidebar.
  Main (success) Flow — Bulk Deletion:
  1. The patient triggers DELETE /gdpr/patient/{patient_id}/sessions.
  2. The system iterates over all session files for that patient ID.
  3. Each file is permanently deleted.
  4. The system returns the count of deleted sessions.
  Alternate Flow:
  3a. If the session file is not found, the system returns 404.
Post Condition:
  The session data is permanently and irrecoverably removed from the
  server. EMR source files are never affected — only chat logs are
  deleted. The system has no ability to recover deleted sessions.

Use Case Number:
  UC-13
Use Case Name:
  Context Compaction (Data Minimisation)
Overview:
  When a conversation grows long, the system automatically summarises
  older conversation turns to reduce token usage and limit the amount
  of personal data held in memory, fulfilling GDPR Art. 5(1)(c) — Data
  Minimisation. Two strategies are available: LLM-based compaction
  (Gemini) and deterministic compaction (regex-based).
Actors:
  System (automatic, triggered during UC-03 message processing)
Pre condition:
  The conversation history must have more than 6 messages AND the
  estimated token count must exceed 3,000 tokens.
Flow:
  Main (success) Flow:
  1. During UC-03 message processing, the system calls
     needs_compaction(messages) which runs estimate_tokens().
  2. If the threshold is exceeded, the system calls
     split_for_compaction(messages) to separate old turns
     (to summarise) from recent turns (to preserve verbatim).
  3. Strategy selection:
     a. If using GeminiService: compact_with_gemini() is called.
        Gemini produces a clinically faithful bullet-point summary
        of the older turns, preserving symptoms, medications,
        conditions, and key decisions.
     b. If using HuggingFace/MedGemma: compact_deterministic() is
        called. Regex patterns extract symptoms, medications,
        conditions, and topics without any API call.
  4. The compacted_summary replaces the old turns in the session.
  5. build_llm_history() constructs the LLM input:
     [compacted summary block] + [last 6 raw turns] + [new user message]
  6. The session's compacted_summary field is updated and persisted.
Post Condition:
  The conversation history sent to the LLM is bounded in size. The
  latest 6 turns are always preserved verbatim for conversational
  coherence. EMR data is never included in the compacted summary —
  it is re-loaded fresh from source on each turn.

Use Case Number:
  UC-14
Use Case Name:
  Voice Input (Speech-to-Text)
Overview:
  The patient can speak their medical query instead of typing it. The
  system records audio via the browser's MediaRecorder API, sends it
  to the backend for transcription via the Sarvam AI STT service, and
  populates the transcribed text in the chat input field. This extends
  accessibility to patients with limited literacy or motor impairments.
Actors:
  Patient
Pre condition:
  The patient must have an active chat session (UC-11). The browser
  must have microphone access permission granted. The Sarvam AI STT
  API must be operational.
Flow:
  Main (success) Flow:
  1. The patient clicks the microphone (🎤) button in the chat
     interface.
  2. The browser requests microphone access (if not already granted).
  3. The MediaRecorder API begins recording audio.
  4. The patient speaks their query.
  5. The patient clicks the microphone button again to stop recording.
  6. The recorded audio blob is sent to the backend:
     POST /sessions/transcribe (multipart form data with
     audio file + language code).
  7. The backend forwards the audio to Sarvam AI's speech_to_text()
     API with the selected language code.
  8. Sarvam returns the transcribed text.
  9. The transcribed text is populated in the chat input field.
  10. The patient reviews and sends the message (continues to UC-03).
  Alternate Flow:
  2a. If the browser does not support MediaRecorder or the user
      denies microphone access, the microphone button is disabled.
Post Condition:
  The transcribed text is placed in the input field, ready for the
  patient to review and send as a regular chat message.

--------------------------------------------------------------------------------

Use Case Number:
  UC-15
Use Case Name:
  Multi-Model LLM Selection
Overview:
  The patient can select which LLM backend processes their queries at
  runtime. The system supports three models: Gemini (cloud, highest
  quality), HuggingFace Qwen (local, lightweight), and MedGemma
  (medical-specialised). The model selection is passed as a parameter
  in each chat request, allowing per-message model switching without
  session interruption.
Actors:
  Patient
Pre condition:
  The patient must have an active chat session (UC-11). At least one
  LLM backend must be configured with valid credentials or model weights.
Flow:
  Main (success) Flow:
  1. The patient selects a model from the dropdown in the chat
     interface (options: "Gemini", "HuggingFace", "MedGemma").
  2. The patient types and sends a message.
  3. The system includes the selected model name in the ChatMessage
     request body (model field).
  4. The SessionsRouter reads the model field and routes the request
     to the corresponding service:
     a. "gemini" → GeminiService.chat()
     b. "huggingface" → HuggingFaceService.chat()
     c. "medgemma" → MedGemmaService.chat()
  5. The selected service generates a response using its underlying
     model, with the RAG-assembled system prompt and GDPR disclaimer.
  6. The response includes model_name in the ChatResponse so the
     patient can see which model answered.
  Alternate Flow:
  4a. If the selected model is not available (e.g., missing API key
      or model weights not loaded), the system falls back to the
      default model (Gemini) and logs a warning.
Post Condition:
  The patient receives a response from the selected LLM. The model
  name is displayed in the response metadata. The patient can switch
  models for the next message without creating a new session.

3.1 Class Descriptions
-----------------------

Class 1: ChatMessage (Pydantic Model)
  State:
    - message: str                   — User's chat text
    - patient_id: Optional[str]      — Patient identifier (default "patient101")
    - model: Optional[str]           — LLM model selection ("gemini", "medgemma", "huggingface")
    - language: Optional[str]        — Target language code (e.g. "en-IN", "hi-IN", "ta-IN")
    - audio_requested: Optional[bool] — Whether TTS audio is requested
    - emr_consent: Optional[bool]    — GDPR Art. 5(1)(a) explicit consent flag
    - store_history_consent: Optional[bool] — GDPR Art. 5(1)(e) storage consent flag
  Behavior:
    - Pydantic validation on all fields

Class 2: ChatResponse (Pydantic Model)
  State:
    - response: str                  — LLM-generated response text
    - input_tokens: int              — Token count for input
    - output_tokens: int             — Token count for output
    - total_tokens: int              — Total token usage
    - model_name: str                — Name of model used
    - audio_content: Optional[str]   — Base64-encoded TTS audio
    - language: Optional[str]        — Response language code
    - emr_fields_used: List[str]     — GDPR Art. 15 evidence (fields accessed)
    - was_compacted: Optional[bool]  — Whether context compaction occurred
  Behavior:
    - Pydantic validation on all fields

Class 3: ChatSession (Pydantic Model)
  State:
    - id: str                        — UUID session identifier
    - patient_id: str                — Patient identifier
    - title: str                     — Session title (from first message)
    - created_at: str                — ISO timestamp of creation
    - messages: List[SessionMessage] — Conversation history
    - compacted_summary: Optional[str] — Rolling clinical summary from compaction
    - emr_consent: bool              — GDPR Art. 5(1)(a) consent
    - store_history_consent: bool    — GDPR Art. 5(1)(e) consent
    - expires_at: Optional[str]      — Auto-deletion datetime (30 days)
  Behavior:
    - Pydantic validation; model_copy() for metadata-only persistence

Class 4: SessionMessage (Pydantic Model)
  State:
    - role: str                      — "user" or "assistant"
    - content: str                   — Message text
    - timestamp: str                 — ISO timestamp
    - emr_fields_used: List[str]     — Per-message GDPR Art. 15 evidence log
  Behavior:
    - Pydantic validation

Class 5: GeminiService
  State:
    - api_key: str                   — Gemini API key from environment
    - client: genai.Client           — Google GenAI client
    - model_name: str                — "gemini-2.5-flash-lite"
  Behavior:
    - chat(message, patient_id, history, ...) → dict  — Sends query to Gemini with system prompt, compacted history, and GDPR disclaimers
    - extract_clinical_data(raw_text) → (str, list)   — Extracts structured clinical data from raw EMR text
    - compact_history(messages) → str                  — Summarizes conversation history using Gemini for data minimisation
    - get_patient_data(patient_id) → str               — Reads patient EMR JSON file

Class 6: HuggingFaceService
  State:
    - model_name: str                — "Qwen/Qwen2-0.5B-Instruct"
    - tokenizer: AutoTokenizer       — HuggingFace tokenizer (lazy loaded)
    - model: AutoModelForCausalLM    — HuggingFace model (lazy loaded)
  Behavior:
    - chat(message, patient_id, history, ...) → dict — Generates response using local Qwen model with context compaction
    - get_patient_data(patient_id) → str             — Reads patient EMR JSON file

Class 7: MedGemmaService
  State:
    - model_name: str                — "google/medgemma-1.5-4b-it"
    - processor: AutoProcessor       — MedGemma processor (lazy loaded)
    - model: AutoModelForImageTextToText — MedGemma model (lazy loaded)
  Behavior:
    - chat(message, patient_id, history, ...) → dict — Generates response using MedGemma with clinical context
    - get_patient_data(patient_id) → str             — Reads patient EMR JSON file

Class 8: SarvamService
  State:
    - api_key: str                   — Sarvam API key from environment
    - base_url: str                  — "https://api.sarvam.ai"
  Behavior:
    - translate(text, source_lang, target_lang) → str — Translates text with automatic chunking (Mayura:v1, 1000 char limit)
    - text_to_speech(text, language_code) → str       — Converts text to speech with chunking (Bulbul:v3, 500 char limit)
    - speech_to_text(audio, language_code) → str      — Transcribes audio via Sarvam STT

Class 9: EmbeddingIndex
  State:
    - cui_list: list[str]            — List of all CUI identifiers
    - index: faiss.IndexFlatIP       — FAISS inner-product index over 407K CUI embeddings
    - cui_names: dict[str, str]      — Human-readable CUI name lookup
    - tokenizer: AutoTokenizer       — SapBERT tokenizer
    - model: AutoModel               — SapBERT encoder
  Behavior:
    - encode(text) → ndarray         — Encode text to 768-dim SapBERT vector
    - search(query_vec, top_k) → list — Search FAISS for nearest CUIs
    - get_name(cui) → str            — Get human-readable name for a CUI

Class 10: KnowledgeGraph
  State:
    - _graph: nx.DiGraph             — NetworkX directed graph of SNOMED relations
  Behavior:
    - neighbors(cui, allowed_relations) → list — Get direct neighbors with optional relation filtering
    - has_node(cui) → bool           — Check if CUI exists in graph
    - num_nodes → int                — Node count property
    - num_edges → int                — Edge count property

Class 11: TermExtractor
  State:
    - tokenizer: AutoTokenizer       — Qwen 2.5 tokenizer
    - model: AutoModelForCausalLM    — Qwen 2.5 1.5B model
  Behavior:
    - extract(query) → list[str]     — Extracts normalized medical terms from natural language

Class 12: SafetyResult (Dataclass)
  State:
    - triggered: bool                — Whether a crisis pattern was detected
    - response: str | None           — Crisis response text (non-None only when triggered)
  Behavior:
    - check_safety(message) → SafetyResult — Regex match against self-harm patterns

Class 13: PipelineResult (Dataclass)
  State:
    - seed_cuis: list[dict]          — Phase 1 CUI results
    - expanded_cui_count: int        — Number of expanded CUIs
    - expanded_cuis: list            — Raw ExpandedCUI list from Phase 2
    - matches: list[MatchedSection]  — Phase 3 EMR matches
    - system_prompt: str             — Phase 4 assembled prompt
    - context_text: str              — Clinical context block
    - total_sections: int            — Sections parsed from EMR
  Behavior:
    - run_pipeline(query, emr_path, index, graph, ...) → PipelineResult — Orchestrates all 4 RAG phases

Class 14: EMRSection (Dataclass)
  State:
    - category: str                  — Section type (lab, symptom, diagnosis, medicine, etc.)
    - text: str                      — Primary text content
    - date: str                      — Date of record
    - value: str                     — Associated value (lab result, vital sign)
  Behavior:
    - parse_emr_file(path) → dict                    — Parse Ruby-style JSON EMR file
    - extract_sections(emr) → list[EMRSection]       — Extract searchable sections from parsed EMR
    - deduplicate_sections(sections) → list[EMRSection] — Deduplicate by category+text

Class 15: MatchedSection (Dataclass)
  State:
    - section: EMRSection            — The matched EMR section
    - matched_cuis: list[str]        — CUIs from expanded set that matched
    - best_score: float              — Highest cosine similarity score
  Behavior:
    - match_sections(sections, expanded_cuis, index, ...) → list[MatchedSection]

6. DESIGN RATIONALE & JUSTIFICATION — EVOLUTION ACROSS ITERATIONS
================================================================================

6.1 LLM Backend: From Monolithic Script → Multi-Model Strategy Pattern
------------------------------------------------------------------------
ITERATION 1 (Prototype):
  - withgemini.py: A single 145-line script. Hardcoded API key. Called the
    Gemini REST API directly via requests.post(). The entire raw EMR JSON was
    dumped into a single prompt as "context."
  - local_app_with_gemma.py: Used Ollama to run Gemma2 9B locally for offline
    scenarios. Same pattern — raw EMR in system prompt, CLI chat loop.
  - Problem: No model switching at runtime. Two completely separate scripts
    with duplicated logic. No token tracking, no structured response.

ITERATION 2 (Web Application):
  - Introduced GeminiService, HuggingFaceService (Qwen 0.5B), and
    MedGemmaService as separate classes behind a common chat() interface.
  - The chat router selects the service based on request.model field.
  - HuggingFaceService uses @lru_cache lazy loading to avoid loading 8GB
    model weights at startup if not needed.
  - Gemini uses the google-genai SDK instead of raw REST calls.
  - Problem: Each service still loaded the full raw EMR and sent it to the
    LLM — no selectivity of which EMR data was relevant.

ITERATION 4 (Final):
  - Retained the Strategy pattern from iteration 2.
  - All three services now accept a system_prompt parameter (from the RAG
    pipeline) instead of loading EMR themselves.
  - Added presidio_analyzer and presidio_anonymizer parameters for PII
    scrubbing before LLM invocation.
  - Added MedGemma as an API Client for Ollama instead of loading locally

WHY THIS MATTERS:
  The Strategy pattern lets us swap LLM backends without touching the
  router or RAG pipeline. Gemini for production (cloud, highest quality),
  Qwen for demos (local, lightweight, no API cost), MedGemma for medical
  domain accuracy. The patient never needs to know which model is running.


6.2 RAG Pipeline: From Raw EMR Dump → 9-Phase Research → Clean 4-Phase Production
-----------------------------------------------------------------------------------
ITERATION 1:
  - extract_data() sent the entire EMR JSON to Gemini with a prompt
    "Extract Diagnosis, Meds, Labs, Symptoms." The LLM did all the work.
  - The extracted JSON was shoved into a system prompt as "context."
  - Problem: Expensive (thousands of tokens per query). No semantic search.
    No knowledge graph. The LLM hallucinated SNOMED IDs.

ITERATION 3 (Research Prototype — standalone rag.py, 1674 lines):
  - Built a 9-phase pipeline as a proof-of-concept:
    Phase 1a: Medical entity extraction via regex/NP-chunking heuristics
    Phase 1b: SapBERT encoding of entities
    Phase 2:  Hybrid CUI search (keyword + FAISS + SapBERT re-ranking)
    Phase 3:  Graph expansion (BFS with ISA throttling, MAX_ISA_FRACTION,
              MAX_ISA_CONSECUTIVE limits, bidirectional edge traversal)
    Phase 4:  Path ranking (semantic + relation weight + ISA penalty)
    Phase 5:  EMR parsing (Ruby-style JSON → Python dict)
    Phase 6:  EMR concept matching (SapBERT + graph intersection hard filter)
    Phase 7:  Temporal filtering (evidence persistence windows per type)
    Phase 8:  Evidence ranking (5-factor weighted: text_sim, graph_support,
              concept_overlap, lab_abnormality, recency)
    Phase 9:  Prompt assembly
  - Problem: Monolithic single-file (1674 lines). CLI-only — no web
    integration. Required 3 pickle files. Over-engineered for real-time
    chat latency (<0.5s target was aspirational).

ITERATION 4 (Production — modularised into app/services/rag/*.py):
  - Simplified to 4 clean phases across 9 focused modules:
    Phase 1: find_cuis() — TermExtractor (Qwen 2.5 LLM, replacing regex
             heuristics from iteration 3) decomposes query into medical
             terms, then SapBERT+FAISS finds seed CUIs.
    Phase 2: expand_cuis() — BFS over SNOMED graph with 18 curated
             DIAGNOSTIC_RELATIONS (vs iteration 3's broader ALLOWED +
             TAXONOMY sets). Simpler, no ISA throttling needed because
             ISA edges were dropped entirely from the allowlist.
    Phase 3: match_sections() — EMR sections encoded via SapBERT, matched
             against expanded CUI set by intersection + cosine threshold.
    Phase 4: assemble_prompt() — Groups matches by clinical category
             (diagnoses first, then symptoms, labs, vitals, meds, history)
             into a structured system prompt.
  - Dropped from iteration 3: path ranking (phase 4), temporal filtering
    (phase 7), evidence ranking (phase 8). These added complexity without
    proportional improvement in chat quality for the web use case.
  - Added: TermExtractor using Qwen 2.5 1.5B LLM for query decomposition,
    replacing iteration 3's regex-based entity extraction — much better at
    mapping colloquial language ("my head hurts") to medical terms
    ("headache").

WHY THIS MATTERS:
  Iteration 3 proved the concept. Iteration 4 made it production-ready by
  modularising the code (9 files instead of one 1674-line monster), dropping
  over-engineered phases, and replacing fragile regex entity extraction with
  a small LLM. The 4-phase pipeline is fast enough for real-time chat
  while still grounding responses in the patient's actual medical records.


6.3 GDPR Compliance: Not Present → Retrofitted → Architecture-Level Integration
----------------------------------------------------------------------------------
ITERATION 1:
  - Zero GDPR awareness. Hardcoded API key in source code. Full EMR sent
    to cloud LLM with no consent mechanism. No session management, no data
    retention policy, no right to erasure.

ITERATION 2:
  - Introduced comprehensive GDPR architecture:
    * ConsentModal in frontend — explicit checkboxes for EMR access
      (Art. 5(1)(a)) and history storage (Art. 5(1)(e)).
    * SessionCreateRequest records consent flags at session creation.
    * Sessions auto-expire after RETENTION_DAYS (30 days) — Art. 5(1)(e).
    * run_retention_cleanup() at startup deletes expired sessions.
    * gdpr_router.py with 4 endpoints:
      - POST /gdpr/consent/{id} — update consent (Art. 5(1)(a,e))
      - GET /gdpr/evidence/{id} — right of access (Art. 15)
      - DELETE /gdpr/sessions/{id} — right to erasure (Art. 17)
      - DELETE /gdpr/patient/{id} — bulk erasure (Art. 17)
    * Context compaction (context_compaction.py) — Data Minimisation
      (Art. 5(1)(c)) by summarising old conversation turns.
    * emr_fields_used per message for evidence panel.
    * GDPR_SYSTEM_SUFFIX appended to all LLM system prompts.
    * Problem: Still sent full raw EMR to LLM when consent was given —
      data minimisation was only applied to conversation history, not EMR.

ITERATION 4:
  - RAG pipeline replaced full EMR dump with selective retrieval — only
    EMR sections relevant to the current query reach the LLM. This is
    true data minimisation (Art. 5(1)(c)) applied to medical records.
  - Added Presidio anonymizer for PII detection and scrubbing before
    LLM invocation.
  - Retained all iteration 2 GDPR infrastructure unchanged.
  - Consent can now be toggled per-request (not just per-session) via
    effective_emr_consent / effective_history_consent logic.

WHY THIS MATTERS:
  GDPR compliance evolved from non-existent (iteration 1) to conversation-
  level (iteration 2: session management, consent, retention, erasure) to
  data-level (iteration 4: RAG-based EMR minimisation, PII anonymisation).
  Compliance is not a feature toggle — it's woven into every layer.


6.4 Context Compaction: No History → Raw History → Dual-Strategy Compaction
---------------------------------------------------------------------------
ITERATION 1:
  - Chat history was a single concatenated string variable (chat_history).
    Every turn appended "USER: ... ASSISTANT: ..." Grow forever until
    the context window exploded.

ITERATION 2:
  - Introduced context_compaction.py with two strategies:
    * compact_with_gemini() — LLM-based summarisation (high quality,
      clinically faithful, costs one extra API call).
    * compact_deterministic() — Regex-based keyword extraction for symptoms,
      medications, conditions, and topics (zero cost, works offline).
  - Trigger: estimate_tokens() > 3000 AND len(messages) > 6.
  - build_llm_history() injects the compacted summary as a synthetic
    "assistant" message block, then appends the last 6 raw turns.

ITERATION 4:
  - Carried forward from iteration 2 unchanged.
  - compacted_summary now stored in ChatSession and persisted across
    page refreshes (if history consent is given).
  - Gemini compaction preferred when using GeminiService; deterministic
    compaction used for HuggingFace/MedGemma.

WHY THIS MATTERS:
  Without compaction, the 11th message in a conversation would exceed
  most local model context windows (Qwen 0.5B: 32K tokens). The dual
  strategy ensures the best available approach is always used, while
  GDPR compliance is maintained — EMR data is never stored in the summary.


6.5 Safety Guardrail: Not Present → Added in Iteration 4
----------------------------------------------------------
ITERATIONS 1–3:
  - No safety mechanism. A patient expressing suicidal ideation would have
    their message processed by the LLM, which might generate an unsafe or
    inappropriate response.

ITERATION 4:
  - Added safety.py with check_safety() — a pre-LLM regex filter that
    matches 13 compiled patterns for self-harm indicators (e.g. "want to
    die", "suicidal", "end my life", "overdose", "better off dead").
  - When triggered, the LLM is completely bypassed and a hardcoded
    CRISIS_RESPONSE is returned immediately with Indian mental health
    helpline numbers (iCall, Vandrevala Foundation, AASRA).
  - The check runs BEFORE any other processing (RAG, translation, LLM).

WHY THIS MATTERS:
  A medical chatbot that doesn't handle crisis situations is dangerous.
  The regex-based approach ensures zero latency for crisis responses and
  eliminates the risk of an LLM generating a harmful reply. The hardcoded
  response guarantees correct helpline information every time.


6.6 Multilingual & Voice: Not Present → Full Stack in Iteration 2 → Carried Forward
-------------------------------------------------------------------------------------
ITERATION 1:
  - English only. Text only. No translation or voice support.

ITERATION 2:
  - Introduced SarvamService with three capabilities:
    * translate() — Mayura:v1 API for 11 Indian languages (hi-IN, ta-IN,
      te-IN, ml-IN, kn-IN, mr-IN, gu-IN, bn-IN, pa-IN, or-IN, en-IN).
    * text_to_speech() — Bulbul:v3 TTS.
    * speech_to_text() — Sarvam STT.
  - Automatic chunking: translate splits at 1000 chars, TTS at 500 chars,
    both respecting sentence boundaries.
  - /transcribe endpoint for voice input.
  - Frontend language selector dropdown.

ITERATION 4:
  - Carried forward from iteration 2 unchanged.
  - Now works seamlessly with RAG pipeline (RAG runs in English; input
    translated before RAG, output translated after LLM response).

WHY THIS MATTERS:
  India has 22 official languages and hundreds of millions of citizens
  who don't speak English. The translate-before/after approach lets us
  use English-trained LLMs and RAG pipelines while serving patients in
  their native language. Voice support (STT + TTS) extends accessibility
  to patients with low literacy.


6.7 Architecture: Monolith → Client-Server → Modular Layered Architecture
---------------------------------------------------------------------------
ITERATION 1 (Monolith):
  - Two standalone Python scripts. No separation of concerns. Model
    selection, EMR loading, prompt construction, API calls, and chat loop
    all in one file. No web server.

ITERATION 2 (Client-Server):
  - FastAPI backend + Next.js frontend (separate processes).
  - Backend split into routers / services / models layers.
  - 3 routers: chat.py, sessions.py, gdpr_router.py
  - 5 services: gemini, huggingface, medgemma, sarvam, context_compaction
  - Pydantic models for API contracts (ChatMessage, ChatResponse, etc.)
  - JSON file-based session store.

ITERATION 3 (Research, CLI):
  - Single rag.py file (1674 lines) — intentionally monolithic to iterate
    quickly on the RAG algorithm without web integration overhead.

ITERATION 4 (Modular Layered — Final):
  - Merged iteration 2's web application with iteration 3's RAG research.
  - RAG pipeline decomposed into 9 focused modules:
    rag/__init__.py, pipeline.py, embeddings.py, graph.py, emr.py,
    emr_match.py, cui_search.py, term_extractor.py, prompt.py,
    graph_expand.py
  - Added safety.py (crisis guardrail) and presidio_anonymizer.py (PII).
  - Session store extracted to services/session_store.py.
  - Heavy resources (EmbeddingIndex, KnowledgeGraph, TermExtractor) loaded
    once at startup via FastAPI lifespan, passed to pipeline as arguments —
    no global state in the RAG modules.

WHY THIS MATTERS:
  The journey from two scripts to a modular layered system follows sound
  software engineering principles. Each component can be developed, tested,
  and deployed independently. The RAG pipeline has zero coupling to the web
  framework. The session store can be swapped from JSON files to a database
  by changing one module. The safety guardrail can be enhanced (e.g. ML-based
  classification) without touching any other code.
